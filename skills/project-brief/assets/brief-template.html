user flow

what they see, tap, type from the moment they land
how do people get into the app — open to anyone, or do they sign in?

Authentication is the "sign in" part of an app. Pick none if anyone can use it without an account — a public calculator, a landing page. Pick anything else if people have their own data they need to come back to.

The simplest sign-in is the magic link: they type their email, get a one-click link, no password to remember. OAuth means "sign in with Google/GitHub/Apple" — fast for users, but you have to set up an account with each provider.

tech stack

what the code is written in — pick whichever your developer knows, or let claude pick

This is the language the code is written in. It mostly affects who can work on the project and how easy it is to find help online.

If you're not coding it yourself, just let claude decide — claude will pick a sensible default for the kind of project you're building, and you can hand the result to any developer who works with that language.

where the app's data lives — accounts, content, history, settings

When your app saves things — a user's profile, their last search, an order — it needs somewhere to put them. This is the database.

Postgres is the boring-and-safe answer for almost everything. Sqlite is a single file on disk, great for tiny apps. Redis is a fast scratchpad, not a primary store — only pick it if you know you need it. Flat files means no database at all, just files on disk.

If you're not sure, defer — claude will pick what fits the project.
